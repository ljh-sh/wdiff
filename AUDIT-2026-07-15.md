# Source-Level Security Audit — vendored wdiff 1.2.2 + diffutils 3.10

This audit reviews `upstream/wdiff/` and `upstream/diffutils/` in
the `ljh-sh/wdiff` repository as shipped in v0.1.0. The audited
trees are byte-for-byte copies of the upstream GNU tarballs;
no local patches have been applied. The audit was conducted by
reading the C source under `src/`. No fuzzing, no coverage
instrumentation, no formal verification.

## Severity scale

| Level | Meaning |
|-------|---------|
| **HIGH** | Reachable by attacker-controlled input, real impact (file overwrite, code execution, etc.) |
| **MEDIUM** | Reachable, but harder to trigger or limited impact (denial of service, partial corruption) |
| **LOW** | Reachable in narrow scenarios, or low impact (decoder corruption, edge case) |
| **INFO** | Not a bug, recorded for posterity |

## Summary

| # | Level | Area | Title |
|---|-------|------|-------|
| 1 | **HIGH** | exec | `popen("diff", ...)` PATH lookup, mitigated in dist build |
| 2 | **HIGH** | exec | `popen($PAGER, ...)` follows `$PAGER`, documented behaviour |
| 3 | MEDIUM | tempfile | Temp files default to `/tmp` (privacy leak on shared systems) |
| 4 | LOW | ctype | `isspace(int)` on `getc()` return without explicit cast |
| 5 | LOW | alloc | `asprintf` overwrites its input pointer (small leak) |
| 6 | INFO | locale | `isspace()` is locale-dependent (CJK users see different boundaries) |
| 7 | INFO | signal | `setjmp/longjmp` from signal handler (technically UB) |
| 8 | INFO | network | No outbound socket / connect in wdiff or diffutils |
| 9 | INFO | diffutils | Vendored at 3.10 (latest stable) to inherit upstream security fixes |

## Detailed findings

### #1 (HIGH) — `popen("diff", ...)` uses `$PATH` lookup

**Where**
- `src/wdiff.c:1005-1019` `launch_input_program` calls
  `readpipe(DIFF_PROGRAM, ...)` where `DIFF_PROGRAM` is `"diff"`
  by default (`src/wdiff.c:29`).

**Effect**

wdiff invokes the `diff` binary via PATH lookup. If the user's
`PATH` is hostile (e.g. `PATH=/tmp:$PATH` and an attacker has
written `/tmp/diff`), wdiff will run the attacker's binary. The
attacker controls the output stream that wdiff parses as
unified-diff directives (`reformat_diff_output` at `wdiff.c:836`).
No code execution through the parsed diff itself (it's read-only
metadata), but the attacker binary can do anything the calling
user can.

This is the same threat model as `git`, `less`, and every other
program that uses `popen` + PATH lookup. Not unique to wdiff.

**Mitigation in the ljh-sh dist**

`scripts/build.sh` passes `DIFF_PROGRAM` as an absolute path
(`/path/to/dist/bin/diff`) to `configure`, so the compiled
binary never traverses `$PATH` to find `diff` at runtime.
The vendored `diffutils` is built into the same dist archive
and its location is known. This closes the finding for the
dist binary; users who build from upstream source are
responsible for the same mitigation.

### #2 (HIGH) — `popen($PAGER, ...)` is attacker-influenced

**Where**
- `src/wdiff.c:1051-1105` `launch_output_program` calls
  `writepipe(program, NULL)` where
  `program = getenv("WDIFF_PAGER") ?: getenv("PAGER") ?: PAGER_PROGRAM`.

**Effect**

When stdout is a tty AND `-a` autopager is on, wdiff invokes
the user's `PAGER`. The `popen` call goes through PATH lookup.
Documented behaviour of `$PAGER` in many tools (less, git, etc.)
— not a wdiff bug. The ljh-sh dist is not exposing this attack
surface beyond what the user already has via `man` / `git log` /
etc. No code change; documented in `README.md`.

### #3 (MEDIUM) — Temp files default to `/tmp`

**Where**
- `src/wdiff.c:534-567` `create_template_filename` reads
  `$TMPDIR`, falls back to `$P_tmpdir` (`/tmp`), then `/tmp`.
- On most systems `$P_tmpdir` is `/tmp`, so wdiff effectively
  always uses `/tmp`.

**Effect**

Although wdiff uses `mkstemp()` (race-free random suffix), the
predictable directory on a multi-user system means a local
attacker can `inotify /tmp/wdiff*` and see that wdiff is being
run, inferring user activity. Not a security exploit, but a
privacy leak on shared systems.

**Suggested fix**

Patch `create_template_filename` in wdiff to honor `$TMPDIR`
first (already done — line 542), and verify `$TMPDIR` is set
on the build host. Or build a private `$XDG_RUNTIME_DIR` path
into the binary. Deferred to a future build-script patch.

### #4 (LOW) — `isspace()` called on `int` from `getc()`

**Where**
- `src/wdiff.c:374, 388, 403, 489`.

**Effect**

POSIX requires `isspace(c)` to be called with a value
representable as `unsigned char` or `EOF`. `getc()` returns
either `0..UCHAR_MAX` or `EOF` (-1), so the value is always
safe in practice. Compiler-pedantic only.

### #5 (LOW) — `asprintf` overwrites its input pointer (small leak)

**Where**
- `src/wdiff.c:1089-1098` `launch_output_program` does
  `asprintf(&lessenv, "%s %s", LESS_DEFAULT_OPTS, lessenv)`.
  The new value overwrites the old one; the old string leaks.

**Effect**

At most a few bytes per wdiff run. Not a real leak.

### #6 (INFO) — `isspace()` uses `LC_CTYPE`-dependent classification

**Where**
- `src/wdiff.c:374, 388` and similar.

**Effect**

In a non-C locale, characters like `\xa0` (non-breaking space)
may be classified as whitespace. Not a security issue — but a
CJK user with `LC_CTYPE=zh_CN.UTF-8` may see different word
boundaries than expected. Documented behaviour.

### #7 (INFO) — `setjmp/longjmp` from signal handler

**Where**
- `src/wdiff.c:187-208` `signal_handler` and
  `src/wdiff.c:1480` `setjmp(signal_label)`.

**Effect**

Standard async-signal-unsafe longjmp from inside a signal
handler. The signal handler does nothing but set a flag and
re-arm, so the longjmp is the only state mutation. Universally
supported on Linux/macOS for `SIGINT`/`SIGTERM`/`SIGPIPE`.

### #8 (INFO) — No outbound network, no `__attribute__((constructor))`

Verified by `grep -rE 'socket|connect|__attribute__\(\(constructor'`
on the vendored source: 0 hits. wdiff has no network surface
and no auto-running initialisation.

### #9 (INFO) — `diffutils-3.10` is the matching upstream version

We vendor diffutils 3.10 (latest stable) to provide the `diff`
binary wdiff shells out to. The relevant diffutils source is
~30k lines. We do not modify it; we link the standard upstream
library output. diffutils has its own CVE history; we ship the
latest stable to inherit the upstream security fixes.

## Trust model

### What we **do** know

| Trust claim | Evidence |
|-------------|----------|
| Vendored wdiff is byte-identical to upstream 1.2.2 | `git diff` against tarball is empty |
| Vendored diffutils is byte-identical to upstream 3.10 | same |
| wdiff is in Debian `main` (not non-free) | passed Debian license + code review |
| wdiff is in Homebrew core | passed Homebrew review |
| No outbound network in wdiff or diffutils | grep clean |
| No `__attribute__((constructor))` in wdiff | grep clean |

### What we **do not** know (limiting factors)

| Limitation | Implication |
|------------|-------------|
| `popen("diff", ...)` uses PATH (mitigated in dist) | threat model: user runs in a hostile env |
| `popen($PAGER, ...)` follows `$PAGER` | documented behaviour |
| Temp files always in `/tmp` | privacy leak on shared systems |
| wdiff upstream not GPG-signing recent releases | a compromised GNU mirror would not be detected |

## Action plan

1. **Patch `scripts/build.sh` to use an absolute-path
   `DIFF_PROGRAM`** pointing at the bundled `diff` binary in
   the same dist archive (the diffutils static build produces
   `bin/diff` alongside `bin/wdiff`). This closes #1 for the
   dist binary.
2. **Document the `$PAGER` / autopager behaviour** in README
   and Pages. No code change; design choice (#2).
3. **Defer #3 (`TMPDIR`)** to a future build-script patch that
   optionally uses `$XDG_RUNTIME_DIR` when set.
4. **Inherit diffutils 3.10** (no local patches) and rebase
   vendored copies when diffutils publishes a security release.
